# 心理预约微信小程序竞品调研

调研时间：2026-07-04
调研目标：理解市面心理咨询预约类微信小程序/移动产品的核心页面、业务逻辑、角色分工和学校场景差异，为后续开发学校心理预约小程序做产品准备。

## 1. 调研结论

心理预约产品大体分三类：

1. 商业心理服务平台
   代表：壹点灵、壹心理、简单心理、好心情、松果倾诉、给力心理、此刻心理等。
   典型目标是获客、匹配、成交、复购，功能重心在咨询师展示、服务分类、付费订单、即时文字/语音/视频、测评、内容社区、评价和会员权益。

2. 咨询师/机构预约 SaaS 或模板小程序
   代表：乐享其约、快预约、竹园云/FastAdmin 预约咨询、小程序源码类项目等。
   典型目标是快速搭建预约闭环，功能重心在咨询师管理、时段管理、预约表单、订单管理、通知提醒、核销、简单客户档案。

3. 高校/学校心理咨询预约平台
   代表：华师心理咨询平台、华东师大心理咨询网上预约系统、广东东软“心灵晴空”、智为心理服务平台、智慧校园心理咨询预约等。
   典型目标不是商业转化，而是身份校验、资源公平分配、风险筛查、咨询伦理、隐私保护、按规则预约和危机转介。

对学校版小程序来说，不能照搬商业平台的“咨询师货架 + 评价 + 价格 + 下单”逻辑。更合适的产品骨架是：

- 学生身份认证 + 知情同意 + 隐私说明作为前置门槛。
- 首页突出“预约咨询、紧急求助、我的预约、服务须知”，弱化营销。
- 预约流程优先“选校区/咨询方式/时段 + 简短诉求 + 风险筛查”，可选择咨询师但不应过度消费化。
- 管理端必须强于学生端：排班、规则、爽约、危机标记、转介、保密权限、统计报表、审计日志。
- 咨询记录与心理健康信息属于高敏感数据，必须做最小化采集、分级权限、加密、审计、知情同意和保密例外说明。

## 2. 市场产品分层

### 2.1 商业心理服务平台

常见页面：

- 首页：推荐服务、热门问题、咨询师入口、心理测评、情绪倾诉、AI 陪伴、科普内容、活动 banner。
- 服务分类页：情绪压力、亲密关系、亲子教育、职场压力、睡眠、个人成长、精神心理问诊等。
- 咨询师列表页：头像、姓名、资质、擅长、从业年限、价格、咨询方式、可约状态、评分/评价、标签。
- 咨询师详情页：详细履历、受训背景、咨询时长、擅长人群、咨询流派、服务须知、用户评价、可约时间。
- 下单预约页：选择咨询方式、日期时段、填写诉求、优惠券/余额、支付。
- 我的订单页：待咨询、待确认、进行中、已完成、取消/退款、评价、复约。
- 测评/内容页：量表测评、报告解读、心理科普、直播课程、公开课。
- 隐私/安全页：隐私政策、安全锁、订单保护、咨询协议。

典型业务逻辑：

- 先用内容、测试、低门槛倾诉承接用户，再引导到付费咨询。
- 咨询师是核心商品，平台通过资质筛选、标签、评价、价格和响应速度帮助用户决策。
- 服务方式多样：文字、电话、视频、线下、即时倾诉、长期咨询方案。
- 预约和订单强绑定，通常存在支付、优惠券、退款、售后、平台客服。
- 复购逻辑明显：再次预约、长期方案、成长档案、测评追踪、会员/课程。

可借鉴点：

- 咨询师信息透明：资质、擅长、风格、受训背景。
- 预约前评估：通过测评或简短问卷帮助用户确认服务类型。
- 预约后提醒：状态通知、开始前提醒、改约/取消指引。
- 隐私安全感：安全锁、匿名表达、咨询协议。

学校版应避免：

- 价格、销量、排名、营销化评价。
- 过度展示咨询师“人气”导致资源倾斜。
- 鼓励学生公开评价咨询师，容易破坏咨询关系和隐私。
- 把高风险学生当普通订单处理。

### 2.2 咨询师/机构预约 SaaS

常见页面：

- 机构首页：机构介绍、服务项目、咨询师团队、预约入口、联系方式。
- 咨询师列表/详情：头像、简介、擅长领域、服务时间、服务地点。
- 日期时段页：按咨询师或房间展示可约时段。
- 预约表单：姓名、电话、咨询类型、首次咨询、问题描述、隐私协议勾选。
- 我的预约：预约详情、状态、取消、预约码/核销码。
- 管理后台：咨询师管理、服务项目、时间段、特殊不可约时间、订单列表、导出。
- 咨询师端：编辑资料、设置排班、查看预约名单、处理订单、核销。

典型业务逻辑：

- 机构先配置咨询师、服务、场地和可预约时段。
- 用户选择咨询师/服务/时段并提交表单。
- 系统生成预约单，通知用户、咨询师和管理员。
- 线下到访可用预约码核销，后台统计订单。
- 有些产品支持短信、公众号/小程序订阅消息、视频咨询。

可借鉴点：

- “后台自定义时段 + 特殊不可约时间”对学校非常关键。
- “预约码/签到码/核销”可改造成学校到场签到。
- “预约记录导出”可用于中心工作量统计，但导出权限必须严格限制。

学校版应避免：

- 使用泛行业预约模板时忽略心理咨询特有的知情同意、保密例外、危机干预。
- 只做预约，不做风险筛查、爽约管理、转介与保密权限。

### 2.3 高校/学校心理预约平台

常见页面：

- 登录/绑定：微信登录、学号/统一身份认证、学校/校区绑定。
- 首页：个体咨询预约、线上倾诉、朋辈辅导、团体辅导、心理测评、心理热线、服务须知、我的预约。
- 来访须知/知情同意：服务边界、保密原则、保密例外、次数限制、迟到/取消规则、危机说明。
- 预约入口：选择校区、咨询方式、咨询类型、时段、咨询师或偏好。
- 信息登记：姓名、学号、学院专业班级、电话、性别、紧急联系人或辅导员信息。
- 简短诉求：主要困扰、希望改善方面、是否首次咨询、紧急程度。
- 风险筛查：自伤/伤人想法、近期危机事件、是否正在精神科就诊或服药。
- 知情同意签署：预约前必须确认。
- 预约成功页：时间、地点/会议方式、签到规则、取消规则、热线。
- 我的预约：待咨询、已预约、请假/取消、已完成、爽约、历史咨询。
- 签到页：扫码签到、提前签到窗口、迟到判定。

典型业务规则：

- 服务对象通常限定为本校在籍学生，或特定开放对象。
- 咨询通常免费，每次 40-50 分钟，部分家庭咨询或团体咨询时长更长。
- 一般要求提前 24 小时预约，取消也要求提前一天或一个工作日前完成。
- 常见限制：每周 1 次、每学期 6-8 次、迟到 15 分钟以上自动取消、爽约累计达到阈值后限制后续预约。
- 保密原则前置展示，同时明确生命风险、法律要求等保密例外。
- 高风险或重性精神障碍场景应引导到医院/危机热线/学校危机干预流程。
- 部分平台会自动匹配咨询师，或让学生增加可用时段后排队等待。

## 3. 关键页面功能清单

### 3.1 学生端

建议底部 Tab：

- 首页
- 预约
- 消息/通知
- 我的

首页模块：

- 紧急求助入口：心理危机热线、校内值班电话、线下地址。
- 预约咨询入口：个体咨询、线上咨询、朋辈咨询、团体辅导。
- 服务须知：预约规则、保密原则、咨询边界。
- 心理测评：可选，MVP 可先不做复杂量表。
- 科普内容：可选，适合后续运营。
- 我的下一次预约：若存在预约，应在首页直接展示。

预约流程：

1. 身份确认：读取/绑定学号、姓名、学院、班级、手机号。
2. 阅读并确认来访须知和知情同意。
3. 选择服务类型：个体咨询、线上咨询、朋辈支持、团体报名。
4. 选择校区/咨询方式/时间偏好。
5. 填写预约表单：困扰类型、简述、紧急程度、首次咨询、可联系时间。
6. 风险筛查：自伤/伤人意念、近期危机、精神科就诊、用药等。
7. 选择或匹配咨询师：学校版建议支持“系统匹配”，减少学生选择压力。
8. 确认预约：展示时间、地点、咨询师、取消规则。
9. 订阅提醒：引导订阅预约成功/变更/开始前提醒。
10. 预约成功：展示到访说明、签到方式、热线和取消入口。

我的预约：

- 当前预约卡片：时间、地点、咨询师、状态、签到/取消。
- 历史记录：只展示必要信息，不展示完整咨询内容。
- 状态流转：待匹配、待确认、预约成功、已提醒、已签到、已完成、已取消、已爽约、已转介。

### 3.2 咨询师端

核心页面：

- 今日预约：按时间排序展示，含学生简称/编号、状态、咨询室、风险提示。
- 预约详情：预约表单、知情同意状态、历史次数、迟到/爽约记录。
- 排班管理：固定排班、临时停诊、特殊不可约时间、最大接待量。
- 咨询记录：仅咨询师本人和授权督导可见，区分预约记录和咨询记录。
- 风险处理：高风险标记、转介记录、危机干预记录。
- 督导/复盘：若学校有督导制度，可配置督导可见范围。

咨询师端注意：

- 咨询师不应看到与当前咨询无关的学生全集数据。
- 风险提示要足够醒目，但不能在普通通知中暴露敏感内容。
- 咨询记录不建议放在小程序端做重编辑，优先做管理后台或受控 Web 端。

### 3.3 管理后台

核心模块：

- 学生管理：身份同步、预约资格、黑名单/限制状态、脱敏查询。
- 咨询师管理：资质、角色、校区、擅长、排班权限、状态。
- 排班管理：校区、咨询室、咨询师、时段模板、假期、特殊停诊。
- 预约管理：筛选、改派、取消、请假、爽约、转介、导出。
- 知情同意管理：版本号、签署记录、启用/停用。
- 表单管理：预约问题、风险筛查项、必填规则。
- 危机管理：高风险预约队列、处理记录、值班人员提醒。
- 通知管理：订阅消息模板、短信/公众号通知、失败重试。
- 数据统计：预约量、到访率、爽约率、咨询师工作量、校区负载、等待时长、风险分布。
- 权限与审计：角色权限、访问日志、导出日志、敏感字段查看日志。

后台必须支持的规则配置：

- 提前预约时间：如至少提前 24 小时。
- 取消截止时间：如前一工作日 17:00 前。
- 单次咨询时长：常见 50 分钟。
- 每周/每学期次数限制。
- 迟到阈值：如迟到 15 分钟以上自动取消。
- 爽约阈值：如累计 2-3 次限制预约。
- 咨询师最大接待量。
- 校区/咨询室容量。
- 假期停诊规则。

## 4. 预约状态机

学校版建议采用以下状态：

```text
草稿
  -> 待风险判断
  -> 待匹配
  -> 待确认
  -> 预约成功
  -> 已提醒
  -> 已签到
  -> 咨询完成
  -> 已归档

异常分支：
预约成功 -> 学生取消
预约成功 -> 咨询师取消/管理员改派
预约成功 -> 迟到
预约成功 -> 爽约
待风险判断 -> 高风险待人工处理
高风险待人工处理 -> 危机转介/人工预约/关闭
待匹配 -> 排队等待/补充时段/无法匹配
```

商业平台常见状态：

```text
浏览咨询师 -> 选择服务 -> 下单待支付 -> 已支付待确认
  -> 预约成功 -> 咨询中 -> 已完成 -> 评价/复购
异常：取消、退款、改约、客服介入
```

学校版和商业版最大差别：

- 学校版先判断“是否适合本中心服务”和“风险等级”，不是先促成订单。
- 学校版咨询资源有限，需要公平排队和规则约束。
- 学校版要保留最少必要数据，并且把访问权限切得更细。

## 5. 数据模型草案

后续开发可从这些表/集合开始：

- users：学生基础身份、openid、学号、学院、班级、手机号、状态。
- counselors：咨询师资料、资质、校区、擅长、角色、状态。
- rooms：咨询室、校区、地址、容量、可用状态。
- schedules：咨询师排班模板。
- slots：具体可预约时段，含咨询师、咨询室、开始结束时间、容量、锁定状态。
- appointments：预约主表，含学生、咨询师、时段、状态、咨询方式、取消/爽约字段。
- intake_forms：预约表单，含困扰类型、简述、紧急程度、首次咨询等。
- risk_flags：风险筛查结果、风险等级、处理状态、处理人。
- consent_records：知情同意版本、签署时间、签署人。
- notifications：订阅消息/短信/公众号通知记录。
- attendance：签到记录、签到时间、迟到分钟、核销人。
- consultation_notes：咨询记录，强权限控制。
- referrals：转介记录，医院/热线/校内危机流程。
- audit_logs：敏感访问、导出、修改、管理员操作。
- config_rules：预约规则、次数限制、取消规则、假期规则。

## 6. 学校版 MVP 建议

### 必须做

- 微信登录 + 学号/统一身份认证绑定。
- 来访须知 + 知情同意。
- 预约：选择校区/方式/时段，填写简短诉求和风险筛查。
- 系统匹配或选择咨询师。
- 我的预约：查看、取消、状态提醒。
- 管理后台：咨询师、排班、预约、取消/爽约、统计。
- 咨询师端：查看日程和预约详情。
- 通知提醒：预约成功、变更、开始前提醒。
- 紧急求助：热线、校内值班电话、危机说明。
- 权限与审计：至少区分学生、咨询师、管理员、督导。

### 第二阶段

- 扫码签到和迟到自动判定。
- 自动排队与候补。
- 朋辈咨询、团体辅导报名。
- 心理测评和测评报告。
- 转介流程和危机干预台账。
- 咨询师工作量统计。
- 多校区/多咨询室资源调度。

### 谨慎做

- AI 陪伴：可做情绪记录或科普，不应替代危机识别和专业咨询。
- 用户评价咨询师：学校环境下不建议公开评价。
- 匿名预约：可支持对普通同学隐藏，但系统后台仍需实名，以便危机处理。
- 聊天功能：涉及敏感内容留存，建议先不做或只做“留言倾诉”且有明确响应时限和保密规则。

## 7. 合规与安全重点

心理咨询预约会处理姓名、学号、联系方式、心理困扰、风险筛查、咨询记录等信息，其中心理健康相关信息属于高度敏感数据。建议：

- 最小化采集：预约阶段只收集完成预约和风险判断所需字段。
- 单独同意：对心理健康、风险筛查、咨询记录做明确单独同意。
- 明确保密例外：生命安全风险、伤害他人风险、法律法规要求、重大公共安全风险等。
- 分级权限：咨询师只能看自己的预约；管理员默认看预约调度信息，不默认看咨询记录。
- 脱敏展示：列表中隐藏手机号、学号部分字段，敏感内容默认折叠。
- 加密存储：咨询记录、风险说明、联系方式等应加密或至少字段级保护。
- 审计日志：谁在何时查看、导出、修改敏感数据必须可追踪。
- 数据保留期限：明确预约记录、咨询记录、通知记录保存多久。
- 导出控制：导出需要权限、理由、范围和日志。
- 小程序隐私指引：上线前配置小程序用户隐私保护指引，并在调用隐私接口前获得授权。
